# 抖音电商罗盘榜单采集工程方案

## 1. 文档状态

- 工程：`douyin-compass-collector`
- 当前运行环境：本地 macOS
- Python：3.12
- 默认时区：`Asia/Shanghai`
- 默认任务：`product_hot_sale_all_level3`

本文档只描述当前代码契约。仓库内自动化测试不代表真实账号、真实钉钉机器人、LaunchAgent 或另一台 Mac 已完成外部验收。

## 2. 目标与范围

工程使用独立 Chrome Profile 保存人工登录态，由 Playwright 读取 Cookie 名称白名单内的当前认证状态，再使用同步 `httpx` 请求罗盘结构化接口。

当前业务范围固定为：

- 遍历分类接口返回的所有非汇总一级分类；
- 每次任务请求一次分类树；
- 根据当次响应动态枚举全部三级分类；
- 排除名称为“全部”的节点；
- 忽略四级及更深节点；
- 每个三级分类采集商品实时榜完整 `total`；
- `page_size` 固定为 10，只递增 `page_no`；
- 所有罗盘 API 共用 0.1～0.3 秒随机串行间隔；
- 不实现页面点击采集、自动重试、并发、多账号或多主机。

三级分类数量和每个分类的榜单条数都以当次接口响应为准，不写死分类个数，也不设置商品条数上限。

## 3. 主调用链

```text
读取并严格校验配置
→ 为每个任务建立 CollectionBatch、Manifest 和 batch_id
→ 启动独立持久化 Chrome Profile
→ 校验登录态并读取 Cookie 名称白名单
→ 创建共享的串行 httpx 客户端
→ 每个任务请求一次分类树
→ 按接口顺序遍历所有一级分类并枚举动态三级分类
→ 为每个三级分类建立 category_run_id
→ 按发现顺序逐分类、逐页请求完整榜单
→ raw gzip → SQLite raw 索引 → Manifest 镜像
→ 分类完整榜单校验
→ 正式发布或终结 dry-run 审计
→ 写入 JSONL 并发送一次钉钉批次汇总
→ 手动模式保留 Chrome；Scheduler 模式自动关闭
```

分类树只在一个任务开始时请求一次，不在每个分类或每个分页批次中重复请求。分类树完整响应会作为本机 raw 审计保存，但不会进入仓库 Fixture、普通日志或通知。

## 4. 请求契约

### 4.1 浏览器与认证

- 使用本机正式版 Google Chrome 和独立 `runtime/browser-profile/`；
- 不复用日常 Chrome Profile；
- Cookie 配置只维护名称白名单，值只在进程内从 BrowserContext 读取；
- User-Agent 从当前 Chrome 动态读取；
- 固定页面入口为 `https://compass.jinritemai.com/shop/chance/rank-product`；
- 不依赖页面点击设置筛选条件。

### 4.2 分类树

分类接口：

```text
GET /compass_api/config_center/category/cate_list
```

当前请求保留已经接通的 `level=4`、`scene=9`、`default_cate_to_level=2`。这些参数后续仍需根据真实证据继续裁剪，当前版本不猜测替代值。

分类解析规则：

1. 按 `data.cate_list` 原始顺序遍历全部一级分类；
2. 一级分类 ID 重复或任意被消费层级结构不合法时立即失败并告警；
3. 对每个一级分类只枚举其直接三级分类；
4. 任意层级名称为“全部”的节点都排除；
5. 四级及更深节点不进入榜单采集队列；
6. 发现顺序跨一级分类连续写入 `discovery_order`，后续 SQLite、Manifest、GUI 和 CSV 都沿用该顺序；
7. 批次根字段保持为空，每个 `category_run` 保存真实一、二、三级分类快照。

### 4.3 商品实时榜

榜单接口：

```text
GET /compass_api/shop/product/product_rank/market_hot_sale
```

当前业务参数：

```text
page_no
page_size
industry_id
category_id
brand_type
price_bin
search_info
rank_data_type
begin_date
end_date
date_type
activity_id
```

`industry_id` 和 `category_id` 来自当次分类发现结果。`industry_id` 使用一级分类 ID；`category_id` 按页面级联选择器格式，将二级分类 ID 与三级分类 ID 使用英文逗号拼接。日期在任务开始时按北京时间计算一次，所有分类和分页复用同一个业务日期。

当前任务配置使用 `brand_type=0` 筛选非知名品牌，使用 `price_bin=10001-?` 筛选价格严格大于 10000 且没有上限。已验证的不限值分别为 `brand_type=-1` 和 `price_bin=不限`；其他品牌或价格带值在取得真实请求证据前不开放。实际请求值随批次写入 SQLite 和 Manifest，作为后续网站展示历史筛选条件的快照。

`_lid`、`verifyFp`、`fp`、`msToken`、`a_bogus`、`btm_*` 以及完整浏览器 cURL 不属于默认请求契约。若最小请求失效，只能基于新证据逐项增加必要字段。

## 5. 分页与分类完整性

第一页返回的 `data.page_result.total` 决定当前分类的完整页数：

```text
target_pages = max(1, ceil(total / 10))
```

- `total=0` 时仍只请求并校验第一页，且 `data_result` 必须为空；
- `total>0` 时完整请求全部页，不截断；
- 同一分类后续页的 `total` 必须与第一页一致；
- 非末页必须返回 10 条，末页条数必须与剩余数量一致；
- 响应页码、固定页大小、顶层状态和数据结构都必须通过契约校验；
- 完成后校验条数、商品 ID、排名唯一性以及排名连续性。

每一页只有在 raw gzip、SQLite 索引和 Manifest 三层都成功后，才计入已保存页。不同任务、批次或分类的分页不得拼接。

## 6. 分类失败阈值

可跳过当前分类继续执行的“普通分类失败”仅包括：

- 网络请求错误；
- HTTP 响应错误；
- 已知响应契约错误。

处理规则：

- 第 1～2 个普通分类失败：记录分类级日志和失败材料，继续采集后续三级分类；
- 正式模式在其余分类成功后发布 `partial_success`；
- dry-run 在其余分类校验完成后终结为 `partial_success`，但不正式发布；
- 第 3 个普通分类失败：立即终止任务，不生成正式商品记录或 CSV；
- 登录失效、中止、数据库、Manifest 或其他内部错误立即终止，不适用上述容错。

当前版本不做分类、分页或批次重试。榜单是实时数据，失败页不会在本次或后续运行中补拼。

## 7. SQLite 与状态模型

当前主要表：

- `collection_batches`：任务尝试、模式、统计、终态和发布边界；
- `category_runs`：动态三级分类快照、顺序、页数、条数和失败位置；
- `raw_responses`：分类分页 raw 路径和捕获时间；
- `product_rank_entries`：正式发布的成功分类商品榜单；
- `product_rank_entry_shops`：商品榜单记录与全部店铺的关系；
- `scheduler_checkpoints`：Scheduler 调和检查点。

日志、raw、Manifest、SQLite 和失败材料使用相同 `batch_id`。分类级材料再使用 `category_run_id` 定位，旧的单任务运行标识不属于当前模型。

### 7.1 正式模式

正式模式在发布事务中：

1. 校验 SQLite 中所有分类终态；
2. 为成功分类准备正式商品和店铺记录；
3. 准备完整 CSV 临时文件；
4. 将 CSV 原子替换到正式路径；
5. 提交 `success` 或 `partial_success`、版本、CSV 路径和 `published_at`；
6. 任一发布步骤失败时回滚数据库并补偿删除本次 CSV。

只有成功分类进入正式商品表和 CSV。失败分类仍保留审计状态和本机失败材料。

### 7.2 dry-run

dry-run 不是无状态模式。它保留：

- `collection_batches`；
- `category_runs`；
- `raw_responses`；
- 分类树和分页 gzip；
- Manifest；
- JSONL 和失败材料。

dry-run 不写正式商品和店铺记录，不生成 CSV，不分配版本，并保持 `csv_path=None`、`published_at=None`。因此 dry-run 的 `success` 或 `partial_success` 不能被当成正式发布，也不会阻止 Scheduler 后续正式执行同一计划。

### 7.3 正式发布唯一判据

`published_at IS NOT NULL` 是唯一正式发布判据：

- 包含正式 `success`；
- 包含正式 `partial_success`；
- 排除 dry-run；
- 排除仅进入 `publishing` 但尚未完成的事务；
- GUI 最近 CSV、幂等版本查询和运维判断都不得只看状态文本或文件是否存在。

## 8. 本机文件布局

```text
runtime/
├── browser-profile/
├── data/
│   └── collector.db
├── exports/
│   └── <YYYY-MM-DD>/<task_id>/
│       └── <中文任务名>_<HHMM>[_vN].csv
├── raw/
│   └── <YYYY-MM-DD>/<task_id>/<batch_id>/
│       ├── category-tree.json.gz
│       ├── manifest.json
│       └── categories/<category_run_id>/page-NNN.json.gz
├── artifacts/
│   └── <YYYY-MM-DD>/<task_id>/<batch_id>/
│       ├── failure.json
│       └── <category_run_id>/
│           ├── failure.json
│           └── failure-response.txt
├── logs/
│   └── <YYYY-MM-DD>.jsonl
└── locks/
```

批次级浏览器故障还可能在批次 artifact 目录保存 `failure.png`。HTTP 响应正文最多保存 1 MiB，只进入本机失败材料。

`runtime/` 整体忽略提交。完整响应只保存在 `runtime/raw/`，仓库 Fixture 只允许保存脱敏契约样本。

## 9. CSV 契约

正式 CSV 使用 UTF-8 BOM，并固定为 7 列：

```text
分类
排名
商品
店铺名称
用户支付金额
成交件数
首次上榜
```

规则：

- “分类”输出完整一、二、三级路径；
- 先按 `discovery_order`，再按分类内 `rank` 排序；
- 一个商品一行；
- 多店铺按接口顺序使用 ` | ` 拼接；
- 金额和成交件数只在展示层按平台单位换算；
- 文件按日期和 `task_id` 目录隔离；
- v1 不加版本后缀，后续强制版本使用 `_vN`；
- 通知只发送 CSV 文件名，不发送本机路径。

## 10. Scheduler

- APScheduler 在前台长期驻留，`launchd` 只负责用户级启动和异常拉起；
- 当前 cron 只接受每日固定时分；
- 同一计划时间的多个任务合并为一个 Chrome 批次并严格串行；
- 启动时根据 SQLite 检查点调和当日遗漏；
- 同日且在宽限期内补采，跨天或超过宽限记录 `missed`；
- Chrome/采集锁被占用时记录 `skipped_busy`，不排队；
- 任一非 dry-run 终态都会阻止同一计划自动重跑，包含正式 `partial_success`；
- dry-run 终态不占用正式计划；
- Scheduler 不等待人工输入，任务结束后自动关闭本次 Chrome；
- 重试策略与 Scheduler 规则仍列入后续重新梳理范围。

## 11. PySide6 GUI

`run` 默认打开 GUI，`--no-gui` 显式回退终端模式。GUI 当前支持：

- 正式模式、dry-run 和强制版本；
- 动态分类总数、分类路径、分类序号和分页进度；
- `partial_success` 与普通成功的不同结果文案；
- 安全 JSONL 事件展示和级别过滤；
- 协作式中止；
- 成功或失败后保留 Chrome，人工检查后再关闭；
- 启动和停止 GUI 自己拥有的 Scheduler；
- 对终端或 launchd Scheduler 只读展示，不越权终止；
- 打开本次或最近已发布 CSV；
- 打开输出目录；
- 在采集和 Scheduler 全部停止时清除本地采集数据。

清除操作会删除 SQLite、CSV、raw、artifact 和 JSONL，保留 Chrome Profile、锁、`.env`、配置和备份。当前不实现 macOS 系统通知。

## 12. 钉钉批次汇总

仓库根目录 `.env` 由程序主动加载，同名外部环境变量优先：

```dotenv
DINGTALK_ENABLED=true
DINGTALK_WEBHOOK_URL=<官方自定义机器人 Webhook>
DINGTALK_SECRET=<加签密钥>
```

行为边界：

- GUI、终端、强制模式、dry-run、Scheduler 和 missed 汇总都可发送；
- 每个批次只发送一次 Markdown；
- 不 @ 任何人；
- 使用 HMAC-SHA256 加签；
- 不跟随重定向、不自动重试；
- 只发送来源、模式、批次、时间、任务状态、页数、条数、CSV 文件名或安全错误分类；
- 不发送 Cookie、Token、本机绝对路径、响应正文或异常原文；
- 通知失败只写安全日志并更新 GUI 状态，不改变采集事务或退出码；
- 只有 `notify-test` 会以通知发送结果决定自己的退出码。

真实 Webhook 测试会产生外部消息，必须由人工明确执行 `make notify-test`。

## 13. OSS CSV 上传与签名下载

OSS 默认关闭。开启 `OSS_ENABLED=true` 且所有配置校验通过时，单个任务的真实调用链为：SQLite 与 CSV 正式发布成功 → 上传 `runtime/exports/` 中的 CSV → 生成最长 7 天有效的私有 GET 签名链接 → 链接仅写入同批次钉钉 Markdown。

对象键为 `compass/<business_date>/<task_id>/<batch_id>/<中文 CSV 文件名>`，强制采集使用新的 `batch_id`，不会覆盖历史对象。RAM 专用用户只需目标 Bucket 的 `oss:PutObject` 与 `oss:GetObject`。

- dry-run、失败、中止与跳过任务不上传；
- OSS 上传处于发布边界之后，失败不回滚 SQLite、本地 CSV 或 `success`/`partial_success`；
- 失败只写 `oss_upload_failed` 等安全分类，钉钉保留 CSV 文件名并标记上传失败；
- 签名 URL 是短期 bearer 凭证，不进入 SQLite、Manifest、JSONL 或普通日志；
- 第一版不实现 OSS 自动重试或断电后的补传恢复。

## 14. 常用命令与测试分组

```bash
make app
make run
make run MODE=dry-run
make run MODE=force
make run GUI=no
make run MODE=dry-run GUI=no
make scheduler
make status
make clear-data
make notify-test
make test
make check
make service ACTION=install
make service ACTION=status
make service ACTION=uninstall
```

`make run` 是唯一采集入口：`MODE` 选择 `normal`、`dry-run` 或 `force`，`GUI` 选择桌面窗口或终端。`make test` 运行完整测试；阶段测试文件仍保留在 `tests/` 供开发排查，但不再作为日常 Make 命令。

测试不启动真实 Chrome、不访问罗盘接口、不发送真实钉钉消息，也不注册 LaunchAgent。真实登录、真实采集、通知和系统守护必须单独人工验收。

## 15. 安全与保留

- Cookie 值、Webhook、签名密钥和完整请求头不得进入 Git、日志、Manifest 或通知；
- 原始响应默认保留 30 天；
- 失败材料和 JSONL 默认保留 10 天；
- SQLite 和正式 CSV 长期保留；
- Chrome Profile 含登录凭证，不普通复制、不提交 Git；
- `launchd` plist 不包含业务凭证；
- 本机备份不得上传公开网盘。
