## 2. 项目目标与范围

建设一个 Python + Playwright + httpx + SQLite 的抖音电商罗盘商品榜单采集工程。工程每天按配置时间启动浏览器，通过独立 Chrome Profile 获取正常登录态，再由 `httpx` 串行请求榜单接口获取结构化 JSON，完成校验、持久化和 CSV 导出。

### 2.1 首版目标

首版只交付当前已有真实请求和响应样本的任务：

- 榜单：商品榜单 / 总榜 / 实时
- 行业：食品饮料，`industry_id=13`
- 类目：水饮冲调，`category_id=1000001823`
- 品牌类型：不限，`brand_type=-1`
- 价格带：不限
- 日期策略：当天实时榜单，`date_type=1`
- 最大采集量：200 条
- 默认执行时间：每天北京时间 14:00

### 2.2 通用性边界

工程定位为“抖音电商罗盘榜单采集框架”，支持在 YAML 中配置：

- 多个商品榜单任务；
- 行业、类目、品牌类型、价格带等筛选条件；
- `max_items`；
- cron 和时区；
- 任务启停。

首版不建设：

- 任意网站通用 RPA 平台；
- 可视化拖拽流程；
- 多账号、分布式和多实例调度；
- 店铺、达人、内容等其他榜单实体；
- 企业微信通知；
- 云主机部署。

店铺、达人、内容榜单以后应根据真实响应样本增加独立 Adapter 和结果表，不提前建设包含大量空字段的万能榜单表。

## 4. 采集方案

### 4.1 主路径

已通过真实请求验证当前接口可不携带 `a_bogus`。首版不逆向签名，采用 Playwright 管理登录态、`httpx` 串行请求的主路径：

```text
读取任务配置
→ 启动独立持久化 Chrome Profile
→ 检查登录态并按 Cookie 名称白名单读取目标域 Cookie
→ 从当前 Chrome 动态读取 User-Agent
→ 将 Cookie 值仅注入运行时 `httpx.Client`
→ 根据 YAML 显式构造榜单业务参数
→ 用 `httpx` 串行循环请求榜单分页接口
→ 每页之间随机等待 1～2 秒
→ 保存原始分页响应
→ 解析并校验完整榜单
→ 单事务写入 SQLite
→ 生成当天 CSV
```

首版不实现页面点击分页 Adapter。`httpx` 路径失败时必须立即停止并输出脱敏证据，不得在同一次运行中静默切换为页面点击。只有在 `httpx` 经验证不稳定且经人工确认后，才建设第二个 Adapter。

目标接口的安全固定契约为：

```text
GET /compass_api/shop/product/product_rank/market_hot_sale
```

首版仅发送经真实请求验证的业务参数：

```text
page_no
page_size
industry_id
category_id
brand_type
price_bin
search_info
rank_data_type
begin_date
end_date
date_type
activity_id
```

`_lid`、`verifyFp`、`fp`、`msToken`、`a_bogus` 和 `btm_*` 等动态、追踪或安全参数默认不发送。如最小请求失败，只能根据明确证据逐项补充必需字段，不能直接复制完整浏览器 cURL。

### 4.2 浏览器生命周期

- Scheduler 长期驻留；
- Chrome 只在批次开始时启动；
- 使用独立持久化 Profile；
- 手动 `run` 用于开发调试，无论成功或失败都保留 Chrome，等待用户按 Enter 后正常关闭；
- Scheduler 触发的运行禁止等待键盘输入，任务结束后自动关闭本次 Chrome；
- `login` 命令专门打开并保留有界面窗口，供人工登录；
- 不直接复用日常 Chrome 用户目录；
- 首版不使用无头模式。

建议配置：

```yaml
browser:
  channel: chrome
  headless: false
  profile_dir: runtime/browser-profile
  locale: zh-CN
  timezone_id: Asia/Shanghai
  keep_open_after_manual_run: true
```

### 4.3 页面入口与请求头

- 固定安全页面入口为 `https://compass.jinritemai.com/shop/chance/rank-product`；
- 配置中不保存 `from_page`、`btm_*` 等导航和追踪参数；
- 首版不通过页面设置榜单筛选，YAML 中的 ID 和其他业务参数直接用于构造 HTTP 请求；
- `User-Agent` 从当前 Chrome 动态读取；
- 固定最小请求头仅包含 `Accept`、`Accept-Language`、`Referer`、`User-Agent` 和 `agw-js-conv`；
- 不手工伪造 `sec-ch-ua`、`sec-fetch-*`、`priority` 等浏览器提示头；
- 开发阶段保留 Chrome，用于核对登录态和 Network，但不依赖页面默认请求作为采集入口。

## 5. 任务配置模型

一个任务对应一套完整且固定的榜单筛选组合。任务可独立启停、执行和统计。自动重试为后续 TODO，不属于首版能力。

首版配置示例：

```yaml
scheduler:
  timezone: Asia/Shanghai
  misfire_grace_minutes: 600
  cross_day_backfill: false

browser:
  channel: chrome
  headless: false
  profile_dir: runtime/browser-profile
  locale: zh-CN
  timezone_id: Asia/Shanghai
  keep_open_after_manual_run: true

auth:
  # 只配置 Cookie 名称；Cookie 值必须在运行时从 BrowserContext 读取。
  # 具体候选白名单由第一阶段真实请求验证后固定。
  cookie_names: []

http:
  concurrency: 1
  page_interval_seconds:
    min: 1
    max: 2
  connect_timeout_seconds: 10
  read_timeout_seconds: 30

database:
  path: runtime/data/collector.db

retention:
  raw_response_days: 30
  failure_artifact_days: 10
  log_days: 10
  delete_database_records: false
  delete_exports: false

tasks:
  - id: product_hot_sale_drinks
    enabled: true
    display_name: 水饮冲调商品实时总榜
    schedule: "0 14 * * *"

    rank:
      type: product_hot_sale
      endpoint_path: /compass_api/shop/product/product_rank/market_hot_sale
      rank_data_type: 1
      activity_id: ""

    filters:
      industry:
        id: 13
        name: 食品饮料
      category:
        id: 1000001823
        name: 水饮冲调
      brand_type: -1
      price_bin: 不限
      search_info: ""

    date:
      strategy: today
      date_type: 1

    pagination:
      max_items: 200
```

首版只实现经过真实请求验证的 `today + date_type=1`。其他相对日期策略必须拿到真实请求样本后再实现，不能猜测平台参数语义。

YAML 使用 Pydantic 严格模式，未知字段、拼写错误或未支持的值必须让程序拒绝启动，不得静默忽略。`page_size=10` 为当前真实接口的已确认约束，首版不对任务开放配置；`max_items` 必须是 10 的倍数且不超过 200；`concurrency` 首版只允许值 `1`。

Cookie 名称白名单可提交 Git，Cookie 值不得进入 YAML、源码、日志、Manifest、Fixture 或诊断材料。上述结构示例中的空列表不是可运行配置：第一阶段必须先填入已提供真实请求中的候选 Cookie 名称，且配置模型必须拒绝空白名单。后续只能根据成功证据缩减白名单。

## 6. 调度、批次与失败处理

### 6.1 Scheduler

- APScheduler 长期驻留；
- `launchd` 负责开机启动和异常拉起；
- 默认每天北京时间 14:00；
- cron 和时区均配置化；
- 后续迁移云主机时使用 `systemd` 替换 `launchd`。

### 6.2 延迟补采

- 14:00 正常运行：立即采集；
- Mac 当天休眠或暂时断网：恢复后在误点宽限期内补采；
- 延迟补采仍保留计划时间 14:00，并单独记录真实采集时间；
- 跨天后不补实时榜单，不用第二天数据冒充前一天数据；
- 跨天未执行的任务标记为 `missed`。

### 6.3 批次失败传播

```text
依次执行全部启用任务
→ 单个任务失败后记录日志，不阻塞后续任务
→ 全部启用任务结束
→ 汇总成功和失败任务
→ 发送本次批次通知
```

首版不实现页级重试、限流退避或批次重试。HTTP 429、已识别的业务限流、超时、网络错误、5xx 和其他采集错误都立即结束当前任务尝试。失败任务等待下一次计划执行或人工手动补跑。

已保存的前序分页只用于调试，不得与后续其他运行的分页拼接，不得发布不完整榜单。

### 6.4 登录失效

批次开始前统一预检登录态。失效时：

- 批次状态标记为 `auth_required`；
- 不执行后续任务；
- 不作为普通采集失败重试；
- 写入日志并发送 macOS 系统通知；
- 手动 `run` 保留有界面 Chrome 供开发调试，等待用户按 Enter 后关闭；
- Scheduler 运行不等待人工输入，自动关闭本次 Chrome 并继续驻留；
- 用户通过 `login` 命令重新登录，完成后再手动补跑。

## 7. 分页与完整性校验

当前真实接口已确认 `page_size=10`。程序只递增 `page_no`，不猜测或尝试其他 `page_size`。目标条数和页数计算：

```text
target_items = min(data.page_result.total, task.pagination.max_items)
target_pages = ceil(target_items / 10)
```

任务启动时按 `Asia/Shanghai` 计算一次 `business_date`，全部分页共用同一组日期参数：

```text
begin_date = YYYY/MM/DD 00:00:00
end_date = YYYY/MM/DD 00:00:00
date_type = 1
```

分页过程中不得重新计算业务日期，避免跨零点时混用两天参数。

每个分页响应在落盘前必须校验：

- HTTP 状态码为 2xx；
- 响应可解析为 JSON；
- 顶层 `st == 0`；
- `data.data_result` 为数组；
- `data.page_result.page_no` 等于当前请求页；
- `data.page_result.page_size == 10`；
- `data.page_result.total > 0`；
- `total` 在同一次运行的全部分页中保持一致；
- 非最后一页返回 10 条，最后一页不超过剩余目标条数。

完成全部目标页后必须校验：

- 实际条数等于目标条数；
- 商品 ID 不重复；
- 排名不重复；
- 排名覆盖 `1..target_items`；
- 捕获请求的行业、类目和筛选参数与任务配置一致；
- 接口 `total=0` 时默认判定异常，不发布空榜单。

任务采用全量发布策略：

- 原始分页响应可以边采边保存；
- 所有目标页成功后才提交正式榜单快照；
- 通过单次数据库事务写入；
- 任何一页失败都不生成正式 CSV；
- 失败尝试的原始响应仅用于排查，不与其他运行拼接。

## 8. 数据设计

### 8.1 通用表

建议包含：

- `collection_batches`：计划批次和批次汇总；
- `collection_runs`：每个任务的每次执行尝试；
- `raw_responses`：原始分页响应路径、页码和摘要。

运行状态至少包含：

```text
pending
running
success
failed
missed
auth_required
```

### 8.2 商品榜单表

首版使用独立的 `product_rank_entries`，不建设万能榜单表。主要字段：

```text
id
batch_id
run_id
task_id
business_date
planned_at
captured_at
page_no
rank
product_id
product_name
newly_on_ranking
pay_amount_min_value
pay_amount_max_value
pay_amount_unit
pay_combo_count_min_value
pay_combo_count_max_value
pay_combo_count_unit
```

平台数值不在持久化层自行解释或换算。保存原始 `value + unit`，例如：

```text
pay_amount_min_value = 100000000
pay_amount_max_value = 250000000
pay_amount_unit = price
```

### 8.3 店铺关系

接口中的店铺为 `product_info.shop_list[]`：

- SQLite 保存全部店铺；
- 使用 `product_rank_entry_shops` 子表保存榜单记录与店铺的关系；
- 保持接口原始顺序；
- CSV 仍保持一个商品一行；
- 多个店铺名称使用 ` | ` 拼接；
- 不复制商品行，避免金额和成交件数被重复理解。

```text
runtime/
├── browser-profile/
├── data/
│   └── collector.db
├── exports/
│   └── YYYY-MM-DD/
├── raw/

```

每次运行都有独立 `run_id` 目录，同一天的多次调试、失败和人工重跑不得互相覆盖。`manifest.json` 至少记录：

```text
run_id
task_id
business_date
status
page_size
max_items
api_total
target_items
saved_pages
saved_items
started_at
finished_at
failed_page（失败时）
error_category（失败时）
```

Manifest 不得记录 Cookie、请求头、完整接口 URL 或响应正文。状态流为 `running → success / failed / interrupted`。

原始分页响应和 Manifest 都必须先写入同目录临时文件，完整关闭后再原子替换正式文件。`.tmp` 文件只代表未完成写入，不得参与解析或成功统计。

保留策略：

- SQLite：长期保留；
- CSV：长期保留；
- 原始 `.json.gz`：保留 30 天；
- 失败截图和诊断 JSON：保留 10 天；
- 日志：保留 10 天；
- Chrome Profile：持续保留，除非人工重置。

`runtime/` 必须全部加入 `.gitignore`。

与测试用 Fixture 的边界：

- `runtime/raw/` 保存本机真实完整响应，不提交 Git；
- `tests/fixtures/` 只使用已提供真实响应的脱敏副本；
- 首版不额外建设合成响应 Fixture，待获得真实边界样本后再补充；
- 原附件不直接复制到仓库。

## 10. CSV 展示规则

CSV 是展示层，字段顺序与当前业务需求保持一致：

```text
排名
商品
店铺名称
用户支付金额
成交件数
首次上榜
```

规则：

- 一个商品一行；
- 多店铺使用 ` | ` 拼接；
- `newly_on_ranking` 输出原始布尔语义；
- 支付金额和成交件数按照网页显示规则格式化；
- `unit=price` 的原值除以 100 得到元，`unit=number` 的原值除以 10 得到件数；
- 展示层使用“万/亿”紧凑格式并去除无意义小数 0，最终必须在阶段 2 人工验收时与网页核对；
- SQLite 仍保存接口原始 `value + unit`；
- CSV 不保存商品图片；
- CSV 格式化集中在 `CsvExporter`，不污染采集和持久化模块。

示例：

```csv
排名,商品,店铺名称,用户支付金额,成交件数,首次上榜
1,【超值家庭装】6大包-42小包-（古法调配-手工研磨）山药粉,巧农婆营养食品旗舰店,¥100万-¥250万,1万-2.5万,false
```

## 11. 幂等与版本

- 唯一计划执行标识基于 `task_id + planned_at`；
- 手动 `run` 默认归属到当天任务 cron 中的计划时间，当前任务为北京时间 14:00；
- 已成功批次再次执行时默认跳过；
- 失败后的人工补跑仍可关联到原计划批次，但会生成新的 `run_id`；
- 延迟补采保留原计划时间并记录真实采集时间；
- `--force` 才能创建新的成功版本；
- 强制重采不覆盖历史 SQLite 数据；
- CSV 文件名包含计划时间和版本号。

示例：

```text
product_hot_sale_drinks_1400.csv
product_hot_sale_drinks_1400_v2.csv
```

## 12. 日志、诊断与通知

### 12.1 日志

- 控制台输出易读文本；
- 文件使用 JSON Lines；
- 按天轮转；
- 保留 10 天；
- 每条任务日志带 `batch_id/run_id/task_id/stage`。

业务字段不要求脱敏，但以下认证信息严禁进入日志：

- Cookie；
- Authorization；
- 请求头；
- `sessionid`；
- `msToken`；
- `a_bogus`；
- `verifyFp/fp`；
- 包含上述参数的完整接口 URL。

### 12.2 失败诊断材料

页面定位或交互失败时保存：

- `failure.png`；
- 脱敏 `failure.json`。

HTTP 请求失败时额外保存：

- `failure-response.txt`：仅包含响应 body，不包含请求头、Cookie 和完整请求 URL；
- 文件最大 1 MiB，超过时截断并在 `failure.json` 中标记。

诊断 JSON 只记录安全页面路径或固定接口路径、页面标题、失败步骤、异常类型、HTTP 状态码或脱敏业务错误码、时间、任务和运行 ID。首版默认不保存完整 HTML 和 Playwright Trace，避免落盘敏感内容和大体积资源。

### 12.3 通知

首版仅实现 macOS 系统通知，内容包含：

- 批次计划时间和实际时间；
- 成功、失败任务数；
- 失败 `task_id`；
- 失败阶段和最后错误摘要；
- 日志及诊断材料路径。

企业微信 Webhook 作为后续 TODO，并预留通知 Adapter 接口。Webhook 地址未来必须通过环境变量提供，不能写入 YAML 或 Git。

## 13. CLI 入口

第一阶段只提供：


第一阶段不提前实现 `--force`、`--dry-run`、`scheduler` 和 `status`。

## 17. 首版验收标准

1. `login` 能打开独立 Chrome Profile，人工登录后持久化登录态。
2. `run` 能从 BrowserContext 按 Cookie 名称白名单读取登录态，使用 `httpx` 采集当前水饮冲调商品实时总榜。
3. 不携带 `a_bogus`、`msToken`、`verifyFp/fp`、`_lid` 和 `btm_*` 时，最小请求经真实验收成功。
4. `page_size` 固定为 10，`page_no` 从 1 递增，分页间串行随机等待 1～2 秒。
5. 根据 `data.page_result.total` 和 `max_items` 完成分页，默认最多 200 条。
6. 每页严格校验 HTTP、`st`、`page_no`、`page_size`、`total` 和条数，任何一页异常都不发布正式结果。
7. 每次运行生成独立 `run_id` 目录、gzip 原始响应和脱敏 Manifest，临时文件原子替换正式文件。
8. 手动 `run` 在成功或失败后保留 Chrome，按 Enter 后正常关闭；Scheduler 运行不等待人工输入。
9. 连接超时默认 10 秒，读取超时默认 30 秒，首版不自动重试。
10. HTTP 失败响应 body 仅在 `runtime/artifacts/` 保存，最大 1 MiB，不记录请求头、Cookie 或完整 URL。
11. SQLite 保存排名、商品、完整店铺数组、支付金额、成交件数、首次上榜和内部标识。
12. 原始分页响应保存为 `.json.gz`，并按 30 天策略清理。
13. CSV 列为排名、商品、店铺名称、用户支付金额、成交件数、首次上榜。
14. CSV 展示格式与网页一致，多店铺用 ` | ` 拼接；在拿到真实多店铺 Fixture 前不声称该场景已通过真实契约测试。
15. APScheduler 常驻，默认每天北京时间 14:00 执行，时间支持 YAML 配置。
16. 登录失效时停止整个批次并通知人工登录；普通任务失败不阻塞后续任务。
17. 首版不实现页级重试、限流退避或批次重试，任务失败后等待下一次计划执行或人工补跑。
18. 默认幂等，只有 `--force` 创建新版本。
19. 页面失败时保存截图和诊断 JSON。
20. 日志保留 10 天，认证凭证不进入日志、Manifest、Fixture 或诊断材料。
21. 少量核心契约测试通过。
22. `launchd` 安装和卸载脚本可用。
23. `README.md` 内容完整，能指导另一台 Mac 完成安装和首次运行。
