# 故障处理

## 快速检查顺序

```bash
uv run --frozen python -m compass_collector status
tail -n 100 runtime/logs/$(date +%F).jsonl
./scripts/status_launchd.sh
```

日志只保存安全摘要。需要查看响应正文或截图时，根据 `run_id` 进入 `runtime/artifacts/YYYY-MM-DD/<run_id>/<task_id>/`。

## auth_required

含义：持久化 Profile 没有可用认证状态，或接口判定登录失效。

处理：

1. 停止 Scheduler；
2. 执行 `uv run --frozen python -m compass_collector login`；
3. 在 Chrome 中重新登录；
4. 按 Enter 正常关闭；
5. 先执行一次 `--dry-run`；
6. 再恢复 Scheduler。

不要把浏览器 Cookie 复制到 YAML、日志或聊天中。

## missed

含义：计划执行已经跨天，或同日延迟超过 `misfire_grace_minutes`。

实时榜单不会跨天补采。不要通过修改业务日期把第二天数据伪装为前一天结果。确认机器休眠、关机或 Scheduler 停止的原因后，等待下一个计划时刻。

## failed

先查看 `error_category`：

- `timeout` / `network_error`：检查网络，当前版本不会自动重试；
- `rate_limited`：等待下一计划批次，不要提高并发；
- `http_error` / `invalid_json`：查看受限 `failure-response.txt`；
- `response_contract_error` 或类似契约错误：对比脱敏 Fixture 和真实响应结构；
- `database_error` / `csv_error`：检查磁盘空间、目录权限和 SQLite 完整性；
- `browser_page_error`：查看 `failure.png` 和 `failure.json`。

失败 run 不会发布不完整 CSV。人工补跑前先确认根因，不要把多个 run 的分页手工拼接。

## Chrome Profile 被占用

症状通常是 Chrome 无法启动或 Profile 锁冲突。

1. 确认没有另一个 `login`、`run` 或 `scheduler`；
2. 使用状态脚本确认 launchd 是否正在运行；
3. 正常停止对应进程；
4. 不要在 Chrome 仍运行时手工删除 `Singleton*` 文件；
5. 再执行前台 Scheduler 或登录命令。

## Scheduler 没有执行

1. 查看 `scripts/status_launchd.sh`；
2. 查看当天 JSONL 是否有 `scheduler_started`；
3. 核对 `config/tasks.yaml` 中 `enabled` 和每日 cron；
4. 查看 `status` 是否已有同一计划时间终态；
5. 如果 launchd 反复退出，在终端前台运行 `scheduler` 获取错误摘要；
6. 修改配置后重启 LaunchAgent。

同一计划时间已有任意终态时 Scheduler 都会跳过，这是防止自动重试的设计，不是故障。

## 钉钉通知失败

1. 执行 `make notify-test`；
2. 查看当天 JSONL 中的 `notification_*` 事件和安全 `error_category`；
3. 确认 `.env` 中 `DINGTALK_ENABLED=true`；
4. 确认 Webhook 为官方 HTTPS 自定义机器人地址，且加签密钥与当前机器人一致；
5. 如果凭证曾出现在聊天、终端历史或其他不受控位置，先在钉钉轮换，再更新本机 `.env`。

通知没有自动重试，失败不会改变采集结果。不要把 Webhook、签名或响应正文复制到日志或聊天中。

## launchd 常用操作

```bash
./scripts/install_launchd.sh --dry-run
./scripts/install_launchd.sh
./scripts/status_launchd.sh
./scripts/uninstall_launchd.sh
```

安装脚本依赖当前工程绝对路径和当前 `uv` 绝对路径。移动工程目录或更换 uv 后，应卸载并重新安装，不能继续使用旧 plist。

## SQLite 检查

```bash
sqlite3 runtime/data/collector.db "PRAGMA integrity_check;"
```

返回值应为 `ok`。不要在 Scheduler 运行时直接修改数据库。
