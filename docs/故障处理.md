# 故障处理

## 1. 快速检查顺序

```bash
uv run --frozen python -m compass_collector status
tail -n 100 runtime/logs/$(date +%F).jsonl
./scripts/status_launchd.sh
```

`status` 会展示模式、批次终态、分类成功/失败/总数、页数、条数以及是否存在 `published_at`。正式发布必须以 `published_at IS NOT NULL` 判断，不能只看 `success`、`partial_success` 或 CSV 文件是否存在。

## 2. 定位本机材料

先从 `status` 或 JSONL 找到 `batch_id`。批次材料位于：

```text
runtime/raw/<日期>/<task_id>/<batch_id>/
runtime/artifacts/<日期>/<task_id>/<batch_id>/
```

分类级日志会额外提供 `category_run_id`，对应：

```text
runtime/raw/<日期>/<task_id>/<batch_id>/categories/<category_run_id>/
runtime/artifacts/<日期>/<task_id>/<batch_id>/<category_run_id>/
```

常见文件：

- `category-tree.json.gz`：本次任务唯一一次分类树响应；
- `manifest.json`：批次和全部三级分类状态镜像；
- `page-NNN.json.gz`：当前分类已成功保存的分页；
- `failure.json`：脱敏失败摘要；
- `failure-response.txt`：最多 1 MiB 的响应正文；
- `failure.png`：批次级浏览器故障截图。

不要手工拼接不同 `batch_id` 或 `category_run_id` 的分页。

## 3. auth_required

含义：独立 Chrome Profile 没有可用认证状态，或罗盘接口判定登录失效。

处理：

1. 停止 Scheduler；
2. 执行 `make login`；
3. 在独立 Chrome 中重新登录；
4. 按 Enter 正常关闭登录窗口；
5. 先执行一次 `make run MODE=dry-run`；
6. 检查审计批次后再恢复正式运行或 Scheduler。

不要把 Cookie、Token 或完整 cURL 复制到 YAML、日志、文档或聊天中。

## 4. 分类树异常

每个任务只请求一次分类树，并遍历接口返回的所有一级分类。以下情况会在榜单分页开始前失败：

- 一级分类 ID 重复；
- 分类层级或字段结构不符合契约；
- 过滤“全部”后没有可采集三级分类。

先查看批次 `category-tree.json.gz` 和安全 `error_category`。一级分类名称或 ID 变化时，当前实现会直接使用当次接口快照；榜单请求始终使用当前三级分类所属的真实一级分类 ID。

## 5. partial_success

正式 `partial_success` 表示：

- 只有 1～2 个网络、HTTP 或响应契约类分类失败；
- 其余动态三级分类已经完整采集；
- 成功分类的正式商品记录和中文 7 列 CSV 已发布；
- 失败分类没有进入正式商品表或 CSV；
- `published_at` 非空；
- Scheduler 不会自动重跑同一计划。

先按 `category_run_id` 检查失败分类材料。当前版本不重试实时榜单，是否人工补跑由使用者根据数据时效决定。

dry-run 也可能是 `partial_success`，但其 `published_at`、版本和 CSV 均为空，不属于正式发布，也不会阻止 Scheduler 后续正式执行。

## 6. failed、abandoned 或 interrupted

普通分类失败阈值：

- 第 1～2 个：记录失败后继续下一个三级分类；
- 第 3 个：立即终止任务，不发布正式商品或 CSV。

登录失效、中止、数据库、Manifest 和内部错误会直接终止，不等待失败计数达到阈值。

常见 `error_category`：

- `timeout` / `network_error`：检查网络，当前版本不会自动重试；
- `rate_limited`：等待下一计划或人工决定补跑，不要提高并发；
- `http_error` / `invalid_json`：查看受限 `failure-response.txt`；
- 响应契约错误：对照本机 raw 与仓库脱敏 Fixture；
- 发布或 CSV 错误：检查磁盘空间、目录权限和 SQLite 完整性；
- `interrupted`：确认是否从 GUI 或进程信号主动中止。

失败批次已经保存的页只用于审计，不得与新批次组合发布。

## 7. dry-run 看起来“成功”但没有 CSV

这是预期行为。dry-run 会保留：

- `collection_batches`；
- `category_runs`；
- `raw_responses`；
- 分类树、分页 gzip、Manifest、日志和失败材料。

它不会写正式商品记录或 CSV，不分配发布版本，并保持 `published_at=None`。需要正式文件时重新执行 `make run`，或使用 `make run GUI=no` 在终端采集。

## 8. Chrome Profile 或采集锁被占用

1. 确认没有另一个 `login`、GUI、终端采集或 Scheduler；
2. 使用 `./scripts/status_launchd.sh` 检查后台实例；
3. 正常停止对应进程；
4. 不要在 Chrome 仍运行时删除 `Singleton*` 文件；
5. 再执行登录或采集。

Scheduler 到期时遇到实时占用会记录 `skipped_busy`，不排队、不重试。

## 9. Scheduler 没有执行

1. 查看 `./scripts/status_launchd.sh`；
2. 查看当天 JSONL 是否包含 `scheduler_started`；
3. 核对任务 `enabled` 和每日 cron；
4. 用 `status` 检查同一计划时间是否已有非 dry-run 终态；
5. 注意正式 `partial_success` 也属于已处理终态；
6. 如果 launchd 反复退出，在终端运行 `make scheduler` 获取安全摘要；
7. 修改配置后重启 LaunchAgent。

跨天或超过宽限期的实时任务只记录 `missed`。不要修改业务日期伪造补采。

## 10. GUI 状态或 CSV 按钮异常

- GUI 进度来自动态分类事件，应显示分类总数、序号、路径和页码；
- “最近已发布 CSV”只选择 `published_at` 非空且文件仍存在的记录；
- dry-run 的成功状态不会成为 CSV 按钮目标；
- 正式 `partial_success` 应显示“部分分类失败，成功结果已发布”；
- GUI 只能停止自己启动的 Scheduler，外部 Scheduler 始终只读。

如果数据库记录已正式发布但 CSV 被人工移动，GUI 会跳过该文件并查找更早仍存在的已发布 CSV。

## 11. 钉钉通知失败

1. 执行 `make notify-test`；
2. 查看 JSONL 中 `notification_*` 事件和安全 `error_category`；
3. 确认根目录 `.env` 中 `DINGTALK_ENABLED=true`；
4. 确认 Webhook 是官方 HTTPS 自定义机器人地址；
5. 确认加签密钥与机器人当前配置一致；
6. 凭证曾暴露时先在钉钉轮换，再更新本机 `.env`。

通知没有自动重试，失败不会改变采集结果或退出码。只有 `make notify-test` 会因通知失败返回非零。

## 12. SQLite 检查

```bash
sqlite3 runtime/data/collector.db "PRAGMA integrity_check;"
```

返回值应为 `ok`。不要在采集或 Scheduler 运行时直接修改数据库，也不要通过手工补写 `published_at` 把未完成批次伪装成正式发布。

## 13. launchd 常用操作

```bash
./scripts/install_launchd.sh --dry-run
./scripts/install_launchd.sh
./scripts/status_launchd.sh
./scripts/uninstall_launchd.sh
```

移动工程目录或更换 uv 后，应卸载并重新安装，不能继续使用包含旧绝对路径的 plist。
