# 备份与恢复

## 1. 数据边界

建议长期备份：

- `runtime/data/collector.db`：批次、动态分类、raw 索引、正式商品和 Scheduler 检查点；
- `runtime/exports/`：正式发布 CSV。

按排障需要选备份：

- `runtime/raw/`：分类树、分类分页 gzip 和 Manifest；
- `runtime/artifacts/`：批次与分类级失败材料；
- `runtime/logs/`：安全 JSONL。

默认不要备份或迁移 `runtime/browser-profile/`。其中包含登录凭证和浏览器本地状态，新 Mac 应重新执行 `make login`。

## 2. 备份前准备

先停止 GUI 拥有的 Scheduler、终端 Scheduler 或 LaunchAgent，确认没有采集正在写入：

```bash
./scripts/status_launchd.sh
./scripts/uninstall_launchd.sh
```

如果没有安装 LaunchAgent，只需正常停止前台 Scheduler 并关闭保留的 Chrome。

## 3. SQLite 一致性备份

创建本机备份目录：

```bash
mkdir -p runtime/backups
```

使用 SQLite 在线备份命令，不直接复制仍在使用的数据库文件：

```bash
sqlite3 runtime/data/collector.db ".backup 'runtime/backups/collector-backup.db'"
```

校验：

```bash
sqlite3 runtime/backups/collector-backup.db "PRAGMA integrity_check;"
```

结果必须为 `ok`。

## 4. CSV 与 raw 目录

CSV 按以下路径隔离：

```text
runtime/exports/<YYYY-MM-DD>/<task_id>/<中文任务名>_<HHMM>[_vN].csv
```

备份时必须保留日期和 `task_id` 目录，避免不同任务的相同中文文件名在外部目录再次冲突。

raw 和失败材料使用：

```text
runtime/raw/<日期>/<task_id>/<batch_id>/
runtime/artifacts/<日期>/<task_id>/<batch_id>/<category_run_id>/
```

如果要保留可追溯关系，应一起备份数据库、对应批次 raw 与 artifact，不能只复制零散分页。

## 5. 当前数据库兼容边界

当前动态三级分类版本使用干净的现行数据库基线，核心表包含 `collection_batches`、`category_runs`、`raw_responses`、正式商品表和 Scheduler 检查点。

旧版数据库若缺少动态分类表、当前约束或 `published_at` 发布边界，只能归档查看，不得直接覆盖当前运行库。当前版本不承诺旧 schema 兼容读取，也不提供从历史实验数据自动重建当前批次关系的迁移。

## 6. 恢复步骤

1. 停止 Scheduler、采集和 LaunchAgent；
2. 保存当前 `collector.db` 的额外副本；
3. 对备份执行 `PRAGMA integrity_check`；
4. 恢复到 `runtime/data/collector.db`；
5. 按原目录层级恢复必要 CSV、raw 和 artifact；
6. 执行 `uv run --frozen python -m compass_collector status`；
7. 检查正式记录的 `published_at`、CSV 路径和文件是否对应；
8. 执行一次 `make run MODE=dry-run GUI=no`，验证分类发现、完整分页和 SQLite/raw 审计；
9. 确认无误后再启动 Scheduler 或安装 LaunchAgent。

CLI 启动时会执行当前 Alembic 升级。不要手工编辑业务表、`alembic_version`、版本号或 `published_at`。

## 7. published_at 与恢复判断

`published_at IS NOT NULL` 是正式发布唯一判据：

- 正式 `success` 和正式 `partial_success` 应同时有版本和 CSV；
- dry-run 即使状态为 `success` 或 `partial_success`，其 `published_at`、版本和 CSV 仍为空；
- 只有 CSV 文件但数据库没有 `published_at`，不能视为正式发布；
- 数据库标记已发布但 CSV 丢失时，应从备份恢复原文件，不要创建空文件占位。

## 8. 新 Mac 迁移

只迁移：

- 仓库；
- 通过完整性校验的 SQLite 备份；
- 必要的 `runtime/exports/`；
- 确实需要保留的 raw、artifact 和 JSONL。

不要迁移旧 Chrome Profile。LaunchAgent plist 包含旧机器绝对路径，也不要直接复制，应在新工程目录重新运行安装脚本。

建议顺序：

1. 安装 Chrome、Python 3.12 和 uv；
2. 克隆仓库并执行 `uv sync --frozen`；
3. 恢复数据库和按 `task_id` 隔离的 CSV；
4. 从 `.env.example` 创建本机 `.env`；
5. 执行 `make login` 重新登录；
6. 执行 dry-run 检查审计链路；
7. 人工确认后执行一次正式采集；
8. 最后再决定是否安装 LaunchAgent。

仓库没有声称上述迁移已在第二台 Mac 完成真实验收。
